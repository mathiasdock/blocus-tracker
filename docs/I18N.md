# Internationalization — blocus-tracker

## The golden rule

**Every visible UI string lives in `lib/i18n.js` in BOTH `fr` and `en` blocks. No exceptions.**

```jsx
// ✅ Good
<p>{t("feed.empty")}</p>

// ❌ Bad — hardcoded French
<p>Rien dans le feed</p>
```

If a key is missing in one language, the UI shows the raw key (`feed.empty`) — visible to users, unprofessional.

## Usage

```jsx
import { useI18n } from "../contexts/I18nContext";

function MyComponent() {
  const { t, lang, setLang } = useI18n();
  return <p>{t("nav.chrono")}</p>;
}
```

For relative time strings (e.g. "il y a 3 min" / "3 min ago"), use `timeAgo(iso, lang)` from `lib/format.js`.

## File structure

`lib/i18n.js`:
```js
export const STRINGS = {
  fr: {
    "nav.chrono": "Chrono",
    "feed.empty": "Rien dans le feed — sois le·la premier·e à publier !",
    // ...~465 keys
  },
  en: {
    "nav.chrono": "Timer",
    "feed.empty": "Nothing in the feed — be the first to post!",
    // ...~465 keys
  },
};
```

Both blocks must contain **the same set of keys**.

## Naming convention

`<domain>.<sub-key>` where `<domain>` matches the page or feature:

| Prefix | Meaning |
|--------|---------|
| `common.*` | shared across pages (save, cancel, close, loading…) |
| `nav.*` | navigation labels |
| `login.*` / `signup.*` / `auth.*` | auth pages |
| `dash.*` | dashboard (timer) |
| `plan.*` | planning |
| `stats.*` | stats |
| `friends.*` | friends |
| `msg.*` | messages |
| `groups.*` | study groups |
| `comm.*` | communities |
| `feed.*` | social feed |
| `profile.*` | profile, settings |
| `modal.*` | UserProfileModal |
| `xp.*` | XP, levels, missions |
| `badge.*` | badges |
| `goal.*` | goals |
| `heatmap.*` | study heatmap |
| `hist.*` | history |
| `pwa.*` | PWA install banner |

## Bilingual style guide

- **Verbs**: imperative form in both languages ("Modifier" / "Edit", not "Modification" / "Editing").
- **Punctuation parity**: if FR ends in `…` or `?`, EN does too.
- **Ellipsis**: use `…` (single char), not `...`.
- **Apostrophes**: FR `'` (typographic), EN `'`. The codebase uses both; match the file.
- **Loading states**: FR `"Chargement…"` / EN `"Loading…"`.
- **Empty states**: positive tone ("Rien encore — lance-toi !" / "Nothing yet — let's go!").
- **Time**: FR `"Aujourd'hui"` / EN `"Today"`. FR `"7 derniers jours"` / EN `"Last 7 days"`.
- **Levels**: FR `"Niv."` / EN `"Lv."` (key: `xp.level`).
- **Counts with units**: keep units localized (`"5 min"` works in both, but `"5 hours"` → `"5 heures"`).

## Adding a key efficiently

The `lib/i18n.js` file is **~1005 lines**. Don't `Read` it in full. Use the **`add-i18n` skill** which uses `Grep` to find an insertion anchor and `Edit` for surgical inserts.

Quick procedure:
1. `Grep` for a nearby existing key in the same namespace (e.g. `"feed.editPost"` → find existing `"feed."` keys to identify the FR block location).
2. `Edit` to insert the new FR key after that anchor.
3. The same key is duplicated lower in the EN block. `Grep` again and insert the EN translation.
4. **Always insert in both blocks in the same commit.**

## Anti-patterns

- ❌ `<span>Chargement…</span>` — hardcoded FR
- ❌ Conditional text in JSX: `lang === "fr" ? "Foo" : "Bar"` — put both in `i18n.js`
- ❌ Adding to `fr` only and "doing en later" — never gets done
- ❌ Same key in different namespaces (e.g. `feed.title` AND `nav.feed.title`) — pick one
- ❌ Long keys with spaces: `"my feed title"` — use dots

## When adding a brand-new feature

1. Pick a namespace prefix (existing or new).
2. List all visible strings in the feature.
3. Add all FR keys in one block, then all EN keys in the other — group by feature.
4. Reference them with `t("namespace.key")` in JSX.
5. Test by toggling lang in `/profile` → both languages should read naturally.
